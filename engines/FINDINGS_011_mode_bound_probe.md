### Conflation analysis on `mode-bound`

"Mode" semantically binds to inner-oct archetype-mode-class (SOURCE/FEEDBACK/DESIRE/PATTERN/SYNTHESIS/RESOURCE/ANCHOR/VOID per canon §10). Tet-faces don't carry "mode" semantics at face-level — they carry Father/Mother polarity (per canon §M.5).

Applying Athena residency-binding criterion from planet-aspect-activate council (2026-05-12): function-name containing residency-specific term (here: "mode") pre-binds to one primitive-class flavor. Function-name should name THE FUNCTION (compute), not the residency.

**`mode-bound` likely fails conflation-test on same precedent that retired `axis-bound`/`axis-generate`/`axis-arm-emit`.**

## Counter-argument: keep `mode-bound`?

Mnemosyne-style drift-prevention concern: if we replace `mode-bound` with `triangle-aspect-activate`, we lose information about the archetype-mode-class semantics. Card-readers seeing `function_class='triangle-aspect-activate'` on an inner-oct face card won't know which archetype mode is at play.

BUT — that's substrate-position metadata, not function-class metadata. Card already has substrate-position fields (face_id=F1-F8, mode=SOURCE/etc.). The function-class names the compute; mode-class is position-anchor metadata. No information loss.

Same precedent: `planet-aspect-activate` on a Gaia card doesn't lose info about Gaia's specific axis (Taurus/Venus-Pluto) — those are position fields, separate from function-class.
